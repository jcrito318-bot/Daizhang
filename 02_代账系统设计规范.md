# 代账系统设计规范文档

## 一、系统架构设计规范

### 1.1 整体架构设计

#### 1.1.1 架构模式
采用前后端分离的B/S架构，基于微服务设计理念：
- **前端层**：Vue3 + Element Plus 单页应用
- **API网关层**：统一入口、路由转发、负载均衡
- **业务服务层**：按业务模块拆分微服务
- **数据访问层**：MyBatis Plus + Redis缓存
- **数据存储层**：MySQL主从集群 + 文件存储

#### 1.1.2 分层架构
```
表现层（Presentation Layer）
  ├─ Web前端（Vue3 + Element Plus）
  └─ 移动端（响应式设计）

接口层（API Gateway Layer）
  ├─ 统一认证授权
  ├─ 请求路由
  └─ 限流熔断

业务逻辑层（Business Logic Layer）
  ├─ 账套服务
  ├─ 凭证服务
  ├─ 账簿服务
  ├─ 报表服务
  ├─ 税务服务
  └─ 客户管理服务

数据访问层（Data Access Layer）
  ├─ MyBatis Plus ORM
  ├─ Redis缓存
  └─ 文件存储

数据存储层（Data Storage Layer）
  ├─ MySQL数据库集群
  ├─ MinIO对象存储
  └─ 日志存储
```

### 1.2 模块划分规范

#### 1.2.1 核心业务模块
- **账套管理模块**：账套创建、初始化、配置
- **科目管理模块**：会计科目、辅助核算
- **凭证管理模块**：凭证录入、审核、过账
- **账簿管理模块**：明细账、总账、日记账
- **报表管理模块**：资产负债表、利润表、现金流量表
- **期末处理模块**：结转、对账、结账
- **票据管理模块**：票据采集、识别、归档
- **税务管理模块**：税务计算、申报、预警
- **银行对账模块**：流水导入、智能匹配
- **薪资管理模块**：员工、薪资计算、凭证
- **固定资产模块**：资产、折旧、报表
- **客户管理模块**：客户档案、合同、收款

#### 1.2.2 支撑模块
- **系统管理模块**：用户、角色、权限、日志
- **消息通知模块**：短信、邮件、站内信
- **文件管理模块**：文件上传、下载、存储
- **定时任务模块**：定时结转、定时提醒

### 1.3 服务间通信规范

#### 1.3.1 同步调用
- 使用RESTful API进行服务间调用
- 统一使用JSON格式传输
- 设置合理的超时时间（默认3秒）
- 实现服务降级和熔断机制

#### 1.3.2 异步通信
- 使用消息队列处理异步任务
- 凭证批量导入、报表生成等耗时操作异步处理
- 消息持久化，确保消息不丢失

## 二、数据库设计规范

### 2.1 数据库命名规范

#### 2.1.1 数据库命名
- 使用小写字母，单词间用下划线分隔
- 格式：`项目名_模块名`
- 示例：`daizhang_account`（代账系统账套模块）

#### 2.1.2 表命名
- 使用小写字母，单词间用下划线分隔
- 表名前缀标识模块：`acc_`（账套）、`sub_`（科目）、`vou_`（凭证）
- 示例：`acc_account_set`（账套表）、`sub_subject`（科目表）

#### 2.1.3 字段命名
- 使用小写字母，单词间用下划线分隔
- 主键字段：`id`（BIGINT类型）
- 外键字段：`关联表名_id`，如`account_set_id`
- 时间字段：`create_time`、`update_time`、`delete_time`
- 状态字段：`status`（TINYINT类型）
- 金额字段：`amount`（DECIMAL(18,2)类型）

### 2.2 表结构设计规范

#### 2.2.1 基础字段

```sql
deleted TINYINT DEFAULT 0 COMMENT '逻辑删除标识 0-未删除 1-已删除',
version INT DEFAULT 1 COMMENT '乐观锁版本号'
```

### 2.3 索引设计规范

#### 2.3.1 主键索引
- 所有表使用BIGINT类型自增主键
- 主键字段名统一为`id`

#### 2.3.2 唯一索引
- 业务唯一约束使用唯一索引
- 命名规范：`uk_表名_字段名`
- 示例：`uk_account_set_code`（账套编码唯一）

#### 2.3.3 普通索引
- 频繁查询的字段建立普通索引
- 外键字段必须建立索引
- 命名规范：`idx_表名_字段名`
- 示例：`idx_voucher_id`（凭证ID索引）

#### 2.3.4 组合索引
- 多字段组合查询建立组合索引
- 遵循最左前缀原则
- 将区分度高的字段放在前面

### 2.4 数据完整性规范

#### 2.4.1 实体完整性
- 所有表必须有主键
- 主键不能为NULL
- 主键值必须唯一

#### 2.4.2 参照完整性
- 外键关系在应用层维护
- 删除数据时检查关联关系
- 使用逻辑删除而非物理删除

#### 2.4.3 域完整性
- 字段类型严格定义
- 金额字段使用DECIMAL(18,2)
- 状态字段使用TINYINT
- 时间字段使用DATETIME

## 三、接口设计规范

### 3.1 RESTful API设计规范

#### 3.1.1 URL设计规范
- 使用名词复数形式：`/api/v1/vouchers`
- 使用连字符分隔：`/api/v1/account-sets`
- 避免使用动词：不使用`/api/v1/getVoucher`
- 层级关系清晰：`/api/v1/vouchers/{id}/details`

#### 3.1.2 HTTP方法规范
- GET：查询资源
- POST：创建资源
- PUT：更新资源（全量）
- PATCH：更新资源（部分）
- DELETE：删除资源

#### 3.1.3 状态码规范
- 200 OK：请求成功
- 201 Created：创建成功
- 204 No Content：删除成功
- 400 Bad Request：请求参数错误
- 401 Unauthorized：未认证
- 403 Forbidden：无权限
- 404 Not Found：资源不存在
- 500 Internal Server Error：服务器错误

### 3.2 请求响应规范

#### 3.2.1 请求参数规范
- 查询参数使用驼峰命名：`?accountSetId=1&year=2026`
- 请求体使用JSON格式
- 必填参数使用`@NotNull`注解校验
- 可选参数提供默认值

#### 3.2.2 响应格式规范
统一响应格式：
```json
{
  "code": 200,
  "message": "操作成功",
  "data": {},
  "timestamp": 1718438400000
}
```

错误响应格式：
```json
{
  "code": 400,
  "message": "参数错误",
  "errors": [
    {
      "field": "voucherDate",
      "message": "凭证日期不能为空"
    }
  ],
  "timestamp": 1718438400000
}
```

#### 3.2.3 分页响应规范
```json
{
  "code": 200,
  "message": "查询成功",
  "data": {
    "list": [],
    "total": 100,
    "pageNum": 1,
    "pageSize": 20,
    "pages": 5
  }
}
```

### 3.3 接口文档规范

#### 3.3.1 使用Swagger/Knife4j
- 所有接口必须添加Swagger注解
- 接口描述清晰准确
- 提供示例请求和响应

#### 3.3.2 接口注释规范
```java
/**
 * 创建凭证
 * 
 * @param request 凭证创建请求
```

## 五、安全设计规范

### 5.1 认证授权规范

#### 5.1.1 JWT认证
- Token有效期：2小时
- Refresh Token有效期：7天
- Token存储：LocalStorage
- 请求头格式：`Authorization: Bearer {token}`

#### 5.1.2 权限控制
- 基于RBAC模型实现权限控制
- 菜单权限：控制用户可见菜单
- 按钮权限：控制用户可操作按钮
- 数据权限：控制用户可见数据范围

### 5.2 数据安全规范

#### 5.2.1 数据加密
- 敏感数据加密存储（密码、身份证号等）
- 使用AES-256加密算法
- 加密密钥通过环境变量配置

#### 5.2.2 SQL注入防护
- 使用MyBatis Plus参数化查询
- 禁止拼接SQL语句
- 输入参数严格校验

#### 5.2.3 XSS防护
- 输入内容HTML转义
- 使用CSP内容安全策略
- HttpOnly Cookie防止XSS窃取

### 5.3 接口安全规范

#### 5.3.1 接口限流
- 单用户限流：100次/分钟
- 单IP限流：1000次/分钟
- 使用Redis实现限流计数器

#### 5.3.2 接口签名
- 重要接口使用签名验证
- 签名算法：HMAC-SHA256
- 签名参数：timestamp + nonce + signature

#### 5.3.3 HTTPS强制
- 生产环境强制使用HTTPS
- 证书使用Let's Encrypt免费证书
- 定期更新证书

## 七、性能优化规范

### 7.1 数据库优化

#### 7.1.1 查询优化
- 避免SELECT *，只查询需要的字段
- 避免在WHERE子句中对字段进行函数操作
- 合理使用索引，避免全表扫描
- 分页查询使用LIMIT，避免深度分页

#### 7.1.2 连接池配置
```yaml
spring:
  datasource:
    druid:
      initial-size: 5
      min-idle: 5
      max-active: 20
      max-wait: 60000
```

### 7.2 缓存优化

#### 7.2.1 缓存策略
- 科目列表、凭证字等基础数据使用Redis缓存
- 缓存过期时间：基础数据24小时，业务数据1小时
- 缓存更新策略：数据修改时主动失效缓存

#### 7.2.2 缓存Key命名
- 格式：`项目名:模块名:业务名:唯一标识`
- 示例：`daizhang:subject:list:1`（账套1的科目列表）

### 7.3 接口优化

#### 7.3.1 异步处理
- 耗时操作使用异步处理
- 批量导入、报表生成等操作异步执行
- 使用消息队列解耦

#### 7.3.2 批量操作
- 批量插入使用`saveBatch`方法
- 批量更新使用`updateBatchById`方法
- 批量删除使用IN语句

## 八、日志规范

### 8.1 日志级别
- ERROR：系统错误，需要立即处理
- WARN：警告信息，可能存在问题
- INFO：重要信息，业务流程记录
- DEBUG：调试信息，开发环境使用

### 8.2 日志格式
```
[时间] [级别] [TraceId] [类名.方法名:行号] - 日志内容
```

### 8.3 日志内容
- 记录关键业务操作
- 记录异常堆栈信息
- 记录接口请求和响应
- 禁止记录敏感信息（密码、身份证号等）

## 九、异常处理规范

### 9.1 异常分类
- `BusinessException`：业务异常，可预期的业务逻辑错误
- `SystemException`：系统异常，不可预期的系统错误
- `ValidationException`：参数校验异常

### 9.2 异常处理
```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    @ExceptionHandler(BusinessException.class)
    public Result<Void> handleBusinessException(BusinessException e) {
        log.warn("业务异常：{}", e.getMessage());
        return Result.error(e.getCode(), e.getMessage());
    }
    
    @ExceptionHandler(Exception.class)
    public Result<Void> handleException(Exception e) {
        log.error("系统异常：", e);
        return Result.error(500, "系统繁忙，请稍后重试");
    }
}
```

### 9.3 异常码规范
- 200：成功
- 400-499：客户端错误
- 500-599：服务端错误
- 1000-1999：系统级错误
- 2000-2999：账套模块错误
- 3000-3999：凭证模块错误
- 以此类推...
